# IdeaHub 账号安全联调 Checklist

> 最后更新: 2026-04-11  
> 适用范围: 改密轮换 token / 退出所有设备

本清单专门覆盖当前账号安全体系里最容易出线上回归的两条链路：

- 改密后当前会话领取新 token，旧 token 立即失效
- 退出所有设备后所有现有 token 立即失效

建议至少准备两个独立会话做联调：

- 会话 A：正常浏览器窗口
- 会话 B：无痕窗口或第二浏览器

建议准备一个已启用密码登录的测试账号，并确保 A、B 两个会话都已经登录同一账号。

## 1. 相关实现点

### 服务端

- `POST /api/auth/change-password`
- `POST /api/auth/logout-all`
- JWT 负载携带 `tokenVersion`
- `requireAuth` 会校验 token 中的 `tokenVersion` 与数据库当前值是否一致

### 前端

- 个人中心 `MePage` 提供“修改密码”和“退出所有设备”入口
- 改密成功后，前端会用响应中的新 token 覆盖本地 token
- 退出所有设备成功后，前端会清空本地 token 并跳转登录页

## 2. 前置检查

- [ ] 测试账号 `hasPassword=true`
- [ ] 数据库中该用户存在 `tokenVersion` 字段
- [ ] A、B 两个会话都能正常访问受保护接口，例如 `GET /api/auth/me`
- [ ] 当前前端个人中心已显示“修改密码”和“退出所有设备”入口
- [ ] 后端已启用以下接口：`POST /api/auth/change-password`、`POST /api/auth/logout-all`
- [ ] 鉴权链路已启用 `tokenVersion` 校验，旧 token 不会继续通过 `requireAuth`

## 3. 路径一：改密后轮换当前 token

### 操作步骤

- [ ] 在会话 A 进入个人中心，填写当前密码和新密码，提交“修改密码”
- [ ] 观察接口 `POST /api/auth/change-password` 返回 `200`
- [ ] 确认响应体包含 `ok: true`
- [ ] 确认响应体包含新的 `token`
- [ ] 确认响应体包含最新 `user` 信息
- [ ] 确认前端会调用 `loginWithToken(res.token)` 用新 token 覆盖本地旧 token
- [ ] 提交成功后，会话 A 继续停留登录态，不应被踢回登录页

### 成功判定

- [ ] 会话 A 随后请求 `GET /api/auth/me` 仍返回 `200`
- [ ] 会话 A 可继续访问其他受保护页面或接口
- [ ] 会话 B 在不重新登录的情况下，再次请求任一受保护接口时返回未授权
- [ ] 会话 B 的旧 token 因 `tokenVersion` 不匹配而失效，而不是因为其他偶发错误

### 建议补充验证

- [ ] 用旧密码重新登录应失败
- [ ] 用新密码重新登录应成功
- [ ] 刷新会话 A 页面后仍保持登录态，说明新 token 已被持久化
- [ ] 会话 B 刷新后应进入未登录态或在下一次受保护请求时被清理

### 失败信号

- [ ] 改密成功后，会话 A 自己也立即掉线，通常说明前端没有接住新 token 或后端返回 token 有问题
- [ ] 会话 B 仍可继续访问受保护接口，说明旧 token 未被版本校验拦截
- [ ] `GET /api/auth/me` 每次请求都会导致 token 再次变化，这属于错误行为

## 4. 路径二：退出所有设备

### 操作步骤

- [ ] 保持 A、B 两个会话都已登录同一账号
- [ ] 在会话 A 进入个人中心，点击“退出所有设备”
- [ ] 确认前端出现二次确认提示
- [ ] 确认接口 `POST /api/auth/logout-all` 返回 `200`
- [ ] 确认响应体仅需返回 `ok: true`
- [ ] 确认成功后前端执行本地 `logout()`，清除当前 token 并跳转登录页

### 成功判定

- [ ] 会话 A 在操作完成后立即回到登录页
- [ ] 会话 A 本地 token 已被清除，刷新后仍为未登录态
- [ ] 会话 B 的旧 token 在下一次访问受保护接口时返回未授权
- [ ] 重新登录后可正常获得新 token，账号未被锁死

### 建议补充验证

- [ ] 若会话 B 停留在受保护页面，手动刷新后应被重定向或提示重新登录
- [ ] 退出所有设备后，再次访问 `GET /api/auth/me` 不应返回旧用户信息
- [ ] 退出所有设备不应修改用户资料本身，只应失效现有会话

### 失败信号

- [ ] 会话 A 点击成功后仍留在登录态，通常说明前端未清本地 token
- [ ] 会话 B 长时间还能继续请求受保护接口，通常说明服务端未正确递增或校验 `tokenVersion`
- [ ] 退出所有设备后当前账号无法重新登录，说明流程误伤了登录能力而不只是清理会话

## 5. 联调观察点

- [ ] 服务端日志中能区分“旧 token 版本失效”和普通未登录错误
- [ ] 前端对未授权返回的处理一致，不会出现部分页面残留旧用户态
- [ ] `change-password` 会返回新 token，`logout-all` 不返回新 token，两条链路语义保持清晰
- [ ] `me` 接口保持只读，不负责轮换 token

### 失败项的实际表现

- 改密后旧 token 访问 `GET /api/auth/me`，实际返回 `500`，消息为 `Unauthorized: Session expired. Please log in again.`
- 改密后其他旧 token 访问 `GET /api/auth/me`，实际返回 `500`
- 使用旧密码重新登录，实际返回 `500`，消息为 `Invalid credentials`
- 退出所有设备后，当前旧 token 再访问 `GET /api/auth/me`，实际返回 `500`
- 退出所有设备后，其他旧 token 再访问 `GET /api/auth/me`，实际返回 `500`

### 初步根因备注

- `auth.controller.js` 的本地登录失败场景仍然是先 `res.status(401)` 再 `throw new Error("Invalid credentials")`
- `middleware/error.js` 只看 `err.status` / `err.code`，不会继承 `res.statusCode`，所以普通 `Error` 最终被包装成 `500 SERVER_ERROR`
- `middleware/auth.js` 在捕获到 `AppError` 后又重新 `next(new Error("Unauthorized: ..."))`，把原本应返回的 `401 UNAUTHORIZED` 语义再次降级成了普通 `Error`

### 本次联调结论

- 改密后当前会话保活：通过
- 改密后其他旧会话失效：语义通过，HTTP 状态码未通过
- 退出所有设备后当前会话清理：接口语义通过，旧 token 返回码未通过
- 退出所有设备后其他旧会话失效：语义通过，HTTP 状态码未通过
